# 品質

最後更新：2026-05-22

評估者預設採取懷疑態度。只要任何硬性門檻低於標準，即使實作看起來大致完成，該 sprint 仍視為失敗。

## 評估標準

| 標準 | 門檻 | 好的狀態 |
| --- | ---: | --- |
| 產品深度 | 4/5 | 功能支援真實訓練流程，包含空狀態、載入、錯誤與回訪使用者狀態。 |
| 功能性 | 4/5 | 已同意的 contract 可端到端運作，邊界案例有被測試而不是假設。 |
| 視覺設計 | 3/5 | UI 易掃描、響應式、符合訓練產品情境，沒有泛用模板感。 |
| 程式品質 | 4/5 | 實作遵循本地模式、邊界清楚，避免不必要抽象。 |
| 可維護性 | 4/5 | 未來代理可從 repo-local context 理解、測試並延伸變更。 |
| 可靠性 | 4/5 | 檢查通過，狀態轉換明確，失敗模式有處理。 |

## 硬性門檻

- 沒有驗證紀錄的功能不可視為完成。
- 沒有 handoff 的 active plan 不可視為完成。
- 沒有瀏覽器驗證的 UI 流程不可視為完成。
- 沒有 validation 或 typed schema ownership 的資料邊界不可視為完成。
- 如果重複 review 意見能變成機械化規則，就不應只停留在文字提醒。

## 評估方法

1. 讀 sprint contract 與產品上下文。
2. 執行 `RELIABILITY.md` 中的必要命令。
3. 像使用者一樣操作受影響流程，至少包含一個邊界案例。
4. 檢查程式碼是否有邊界漂移、重複與隱藏假設。
5. 為每個標準評分，並列出未通過門檻的具體修正。

## 評分規則

分數不取平均。最低的硬性門檻分數決定通過或失敗。
